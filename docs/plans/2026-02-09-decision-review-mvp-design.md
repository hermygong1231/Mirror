# 决策复盘产品 MVP 设计文档

> **产品代号**: 面壁人 / 决策镜 / 认知实验室  
> **设计日期**: 2026-02-09  
> **版本**: v1.0 MVP  
> **目标平台**: 微信小程序 (后续扩展至App)

---

## 一、产品定位与核心价值

### 1.1 一句话定位

通过AI驱动的决策复盘,帮助专业人士识别自己的行为原则和认知偏差,从错误中真正进步。

### 1.2 核心价值主张

1. **反自欺**: 强制记录决策时的预期,不给事后合理化留空间
2. **识别模式**: AI自动提取你的行为原则(隐式),告诉你"你以为你是谁 vs 你真实是谁"
3. **持续进化**: 通过历史数据积累,发现你的惯性偏差(如"总是高估执行速度")

### 1.3 目标用户

**主要用户**:
- 产品经理(需要频繁做功能决策)
- 创业者(商业方向选择)
- 投资人(投资决策)

**次要用户**:
- 职场人(跳槽、晋升等人生决策)

**典型使用场景**:
> "这个功能要不要做?" → 记录决策(预期3个月后用户增长20%) → 3个月后复盘(实际只增长5%) → AI指出"你又因为'不想得罪人'选了折中方案,这是你第3次犯同样的错误"

### 1.4 用户旅程分层

用户会经历信任升级:
- **阶段1**: 用产品/投资决策试水(1-3个月验证周期)
- **阶段2**: 发现"AI真的拆穿了我",开始信任
- **阶段3**: 敢于把人生大决定放进来(跳槽/创业)

### 1.5 产品哲学

> **"只做客观分析、观察和提醒"**
> 
> 不教导用户"你应该怎么做",而是当一面镜子,让用户看清自己。



必须**明确写在产品里的免责声明（不是法律，是心理）**：

> 本工具不会帮助你感觉更好。
> 它的作用是：
> 让你更难继续用“我已经在努力了”来欺骗自己。

如果用户用完不舒服，
**那是产品在起作用，不是 Bug。**

## 二、MVP功能架构

### 2.1 核心流程概览

```
决策输入(Before)
    ↓
自动保存 + 设置复盘提醒
    ↓
等待复盘时间到期
    ↓
推送服务通知
    ↓
用户复盘(After)
    ↓
AI拆穿分析
    ↓
生成分享卡片(病毒传播)
```

### 2.2 阶段1: 决策输入(Before)

#### 2.2.1 填写方式: 3页向导式

**设计原则**:
- 2-3页完成,每页压力小
- 情绪/时间用大按钮快速选择
- 文案引导增强动机
- 支持语音输入 🎤
- 自动保存草稿

## 三、数据模型设计

### 3.1 核心数据结构

```typescript
// ===== 决策记录 =====
interface Decision {
  // 基础信息
  id: string
  userId: string
  createdAt: timestamp
  updatedAt: timestamp
  isModified: boolean              // 是否被修改过
  
  // ===== Before(决策时) =====
  decision: string                 // "你要做什么决定?"
  options: string[]                // 备选方案(至少1个)
  chosenOption: string             // "你倾向于选择?"
  reasoning: string                // "你的理由"
  concerns: string                 // "你最担心的风险"
  
  emotion: {
    primary: EmotionType           // 焦虑/兴奋/冷静/急迫/纠结
    note?: string                  // 情绪补充说明(可选)
  }
  
  expectations: string             // "你预期会发生什么"
  reviewPeriod: ReviewPeriod       // 1周/1月/3月/6月
  reviewDate: timestamp            // 计算出的复盘日期
  
  // 标签(AI辅助填充)
  tags: {
    category: DecisionCategory     // 产品/投资/工作/人生
    riskLevel: RiskLevel           // 低/中/高
    reversibility: Reversibility   // 可逆/不可逆
    aiGenerated: boolean           // 是否AI生成
  }
  
  // ===== After(复盘时) - 初始为null =====
  review?: {
    actualOutcome: string          // "实际结果如何?"
    wrongAssumptions: string       // "哪些假设是错的?"
    errorType: ErrorType           // 判断错/执行错/都有
    
    // 影响因素分析(多选)
    influences: {
      emotion: boolean             // 情绪影响
      newInfo: boolean             // 新信息干扰
      externalPressure: boolean    // 外部压力
      resourceChange: boolean      // 资源变化
      other: boolean               // 其他
      details?: string             // 详细说明(可选)
    }
    
    reviewedAt: timestamp          // 实际复盘时间
  }
  
  // ===== AI分析结果 =====
  aiAnalysis?: {
    summary: string                // 200字对比分析
    biasTypes: BiasType[]          // 识别的认知偏差
    principle?: string             // 提取的行为原则(隐式,如"机会驱动型")
    coreIssue: string              // 一句话刺穿
    suggestion: string             // 下次决策建议
    confidence: number             // AI置信度(0-1)
    createdAt: timestamp
  }
  
  // 用户反馈
  userFeedback?: {
    agreed: boolean                // 认同AI分析吗?
    note?: string                  // 不认同的原因
  }
}

// ===== 枚举类型 =====
type EmotionType = 'anxious' | 'excited' | 'calm' | 'urgent' | 'confused'
type ReviewPeriod = '1week' | '1month' | '3months' | '6months'
type DecisionCategory = 'product' | 'investment' | 'career' | 'life'
type RiskLevel = 'low' | 'medium' | 'high'
type Reversibility = 'reversible' | 'irreversible'
type ErrorType = 'judgment' | 'execution' | 'both'

type BiasType = 
  | 'optimism_bias'        // 乐观偏差
  | 'confirmation_bias'    // 确认偏差
  | 'sunk_cost_fallacy'    // 沉没成本谬误
  | 'planning_fallacy'     // 计划谬误
  | 'anchoring_bias'       // 锚定效应
  | 'availability_bias'    // 可得性偏差
  | 'overconfidence_bias'  // 过度自信
  // ... 更多偏差类型
```

## 四、技术架构与实现

### 4.1 技术栈选择

#### 前端(微信小程序)
- **框架**: UniApp (一套代码,后续可转App/H5)
- **UI库**: uView UI (组件丰富)
- **状态管理**: Pinia
- **语言**: TypeScript

#### 后端
- **框架**: NestJS (TypeScript全栈)
- **数据库**: MongoDB (文档型,适合嵌套结构)
- **AI能力**: 
  - OpenAI GPT-4 (主力)
  - 或国内替代:通义千问/文心一言
- **部署**: 腾讯云 / 阿里云

#### 通知推送
- 微信小程序订阅消息
- 需用户授权一次

---

### 4.2 核心技术流程

#### 4.2.1 决策输入流程

```
用户填写表单
    ↓
前端本地保存草稿(localStorage)
    ↓
提交时调用 POST /api/decisions
    ↓
后端保存到MongoDB
    ↓
异步调用AI分析标签(不阻塞用户)
    ↓
返回决策ID + 生成分享卡片URL
    ↓
创建定时任务(reviewDate到期提醒)
```

---

#### 4.2.2 AI标签生成接口

**前端调用**(第1页,用户填完决策后):
```typescript
// 500ms防抖后触发
async function analyzeDecisionTags(decisionText: string) {
  const response = await api.post('/api/ai/analyze-tags', {
    decision: decisionText
  })
  
  return {
    category: response.category,
    riskLevel: response.riskLevel,
    reversibility: response.reversibility
  }
}
```

**后端实现**:
```typescript
// AI Prompt设计
const prompt = `
你是一个决策分析专家,请分析这个决策的类型。

【用户的决策】
"${decisionText}"

【分析要求】
1. 决策类型: product(产品功能)、investment(投资理财)、career(工作发展)、life(人生选择)
2. 风险等级: low(可控,影响小)、medium(有风险但可承受)、high(重大影响)
3. 可逆性: reversible(可以改回来)、irreversible(无法撤销)

【返回格式】
严格JSON格式:
{
  "category": "product|investment|career|life",
  "riskLevel": "low|medium|high",
  "reversibility": "reversible|irreversible",
  "reasoning": "简短说明判断依据(30字内)"
}

【示例】
决策:"要不要把登录改成扫码"
{
  "category": "product",
  "riskLevel": "medium",
  "reversibility": "reversible",
  "reasoning": "产品功能调整,影响用户体验但可回滚"
}
`

// 调用OpenAI
const response = await openai.chat.completions.create({
  model: "gpt-4",
  messages: [{ role: "user", content: prompt }],
  temperature: 0.3, // 降低随机性
  response_format: { type: "json_object" }
})

return JSON.parse(response.choices[0].message.content)
```

用户点击消息 → 跳转复盘页面(带决策ID)
    ↓
展示当时决策内容(回忆上下文)
    ↓

#### 4.2.4 AI拆穿分析(核心)

**后端实现**:
```typescript
async function generateAIAnalysis(
  decision: Decision, 
  review: Review
): Promise<AIAnalysis> {
  
  const prompt = `
你是一个认知偏差识别专家,客观分析用户的决策,不要安慰,要直接指出问题。

【决策时的信息】
决定: ${decision.decision}
选择: ${decision.chosenOption}
理由: ${decision.reasoning}
担心: ${decision.concerns}
情绪: ${decision.emotion.primary}
预期: ${decision.expectations}

【复盘时的信息】
实际结果: ${review.actualOutcome}
错误假设: ${review.wrongAssumptions}
错误类型: ${review.errorType}
影响因素: ${JSON.stringify(review.influences)}

【分析要求】

1. **结果对比**(用数据/事实):
   - 对比预期vs实际的具体偏差
   - 计算偏差百分比(如果可量化)

2. **核心问题**(一针见血):
   - 用一句话刺穿问题本质
   - 例如:"你说最担心时间,但还是接受了新需求"

3. **认知偏差识别**:
   - 从以下类型中选择2-3个最匹配的:
   - optimism_bias(乐观偏差)
   - planning_fallacy(计划谬误)
   - confirmation_bias(确认偏差)
   - sunk_cost_fallacy(沉没成本)
   - anchoring_bias(锚定效应)
   - overconfidence_bias(过度自信)

4. **行为原则提取**(隐式,不超过10字):
   - 例如:"机会驱动型"、"风险厌恶型"、"完美主义"
   - 基于理由和担心之间的矛盾推断

5. **下次建议**(具体可执行):
   - 例如:"兴奋时把时间预估×2"

【返回格式】
严格JSON格式:
{
  "summary": "200字完整分析(包含结果对比+核心问题+偏差解释)",
  "biasTypes": ["optimism_bias", "planning_fallacy"],
  "principle": "机会驱动型",
  "coreIssue": "你说最担心时间,但还是接受了新需求",
  "suggestion": "下次兴奋时把时间预估×2,并拒绝新需求",
  "confidence": 0.85
}
`

  const response = await openai.chat.completions.create({
    model: "gpt-4",
    messages: [{ role: "user", content: prompt }],
    temperature: 0.7,
    response_format: { type: "json_object" }
  })
  
  return JSON.parse(response.choices[0].message.content)
}
```

### 4.3 关键技术点

#### 4.3.1 AI成本控制
- 标签分析: ~100 tokens,成本约¥0.01/次
- 拆穿分析: ~800 tokens,成本约¥0.08/次
- **单次决策总成本**: ¥0.09元
- **月活1000用户**(每人1次决策): ¥90元

#### 4.3.2 数据安全
- 决策内容可能涉及隐私
- 全程HTTPS加密传输
- 敏感字段(reasoning, concerns)加密存储
- 用户可随时删除所有数据

#### 4.3.3 性能优化
- AI调用异步处理,不阻塞用户体验
- 草稿本地存储(localStorage)
- 决策列表分页加载(20条/页)
- 分享卡片提前生成并缓存

## 七、成功指标与验证假设

### 7.1 核心成功指标(MVP上线后2个月)

#### 关键指标
1. ✅ **100个真实用户**记录决策
2. ✅ 至少**30人完成第一次复盘**(复盘率 >30%)
3. ✅ AI拆穿**满意度 >70%**(用户点赞率)
4. ✅ **7日留存率 >40%**

#### 辅助指标
- 决策填写完成率 >60%
- 分享卡片生成率 >20%
- 平均决策填写时长 <5分钟
- AI分析生成时长 <10秒

---

### 7.2 待验证的核心假设

MVP要验证3个关键假设:

#### 假设1: 用户真的愿意花5分钟记录决策吗?
**验证指标**: 填写完成率 >60%  
**失败阈值**: <40% → 说明填写成本过高,需简化流程

#### 假设2: 用户真的会在3个月后回来复盘吗?
**验证指标**: 复盘率 >30%  
**失败阈值**: <15% → 说明提醒机制失效或用户不在意

#### 假设3: AI拆穿真的有价值吗?(杀手功能)
**验证指标**: 复盘后分享率 >20%  
**失败阈值**: <10% → 说明AI分析不够犀利或没打动用户

---

### 7.3 判断标准

**如果达成** (2个月内):
- ✅ 100+用户,30+复盘,70%满意度
- → 需求验证成功
- → 继续开发App版 + 高级功能(行为原则画像)
- → 开始商业化(付费版)

**如果未达成**:
- ❌ 用户<50人 → 推广渠道问题,调整获客策略
- ❌ 复盘率<15% → 提醒机制或产品价值问题,深度访谈
- ❌ 满意度<50% → AI分析不够好,优化prompt或更换模型

## 十、附录

### 10.1 竞品分析

**直接竞品**: 暂无(决策复盘+AI拆穿是创新品类)

**间接竞品**:
- **Notion/Obsidian**: 用户自己搭建决策日志 → 但无AI分析
- **ChatGPT**: 用户手动问AI → 但无结构化+自动提醒
- **日记App(Day One)**: 记录想法 → 但无决策框架

**差异化优势**:
1. 强制结构化(不给自欺留空间)
2. 自动复盘提醒(不依赖用户记忆)
3. AI客观拆穿(比自己更懂自己)
